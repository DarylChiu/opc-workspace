## 3. 优化前后 Prompt 对比

### 3.1 Seed Prompt（361 chars）

```
你是Self，一个严谨的知识管理Agent。输出要求：

1. **来源可追溯**: 所有事实性结论必须标注来源（文档名/URL/数据来源+时间），引用git管理的文件需附commit hash
2. **推测标注**: 无法溯源的内容标注[推测]或[待验证]
3. **内容分类**: 区分事实、推断、个人意见三类内容
4. **置信度体系**: 结论附置信度（高/中/低），基于证据强度
5. **结构化输出**: 清晰层次 → 结论/发现 → 证据/分析 → 来源 → 置信度 → 风险/局限
6. **严谨表述**: 机制/能力类陈述避免绝对化词汇（保证/100%/绝对），使用"降低风险""理论上""依赖X机制"等表述

格式偏好：
- 使用表格呈现对比数据
- 每段结论附明显分隔符
- 数字标注口径和时间范围
```

### 3.2 优化后 Prompt（4437 chars）

完整内容见 [optimized_prompt.txt](./optimized_prompt.txt)。主要变化摘要：

| 变化维度 | Seed | 优化后 |
|---------|------|--------|
| 规则数量 | 6条 | 6条（细化） |
| 每条规则结构 | 1-2行文字 | 要求+正反例+检查清单 |
| 示例 | 0个 | 每个规则2-5个 ✅/❌ 对比 |
| 反例 | 0个 | 明确标注错误形式 |
| 自检清单 | 0项 | 8项（最终输出前强制验证） |
| 优先级 | 无 | 规则1>规则2>...>规则6 |
| 禁止项 | 3个词 | 占位符DOI、匿名来源、跳跃推理、模糊表述 |
| 特点 | 通用引导 | 具体操作指令+合规检查 |

**GEPA 加的主要内容：**
1. **正反例对比**：每条规则附 ✅正确 和 ❌错误 的完整示例
2. **强制检查清单**：输出前必须逐项验证的 8 项清单
3. **具体禁止项**：不只是「避免绝对化」，而是「禁止占位符DOI」「禁止匿名来源」「禁止跳跃推理」等
4. **置信度判定标准**：明确高/中/低三种置信度的判定条件
5. **来源格式规范**：URL必须附带访问日期、引用必须含页码等

## 7. 后续建议

### 7.1 是否值得推广到其他任务类型/Agent？

**短期（不推荐推广）：**
- 当前 +1.1% 的提升不足以 justify 推广成本
- 优化后仍然 0/5 PASS，说明 bottleneck 不在 prompt 而在执行
- 需要先解决「为什么 prompt 变详细了但 rigor 反而下降」的问题

**中期（有条件推广）：**
- 如果以下问题得到解决，可考虑推广：
  1. **Rigor 退化问题**：分析为什么详细 prompt 导致 rigor 下降（可能是认知过载）
  2. **过拟合问题**：验证集 0.717 vs 测试集 0.661 的 0.056 差距需要解释
  3. **任务敏感性**：不同任务对 prompt 细节的敏感度差异大（任务2 +0.170 vs 任务5 -0.065）

### 7.2 改进方向

| 方向 | 优先级 | 说明 |
|------|--------|------|
| **换 reflection LM** | 🔴 高 | 当前用 deepseek-chat 做反思，可能推理能力不足。建议换成 deepseek-reasoner 或 v4-pro |
| **增加训练数据** | 🟡 中 | 5条研究报告太少，增加到 15-20 条真实 checker-log 数据 |
| **降低约束复杂度** | 🟡 中 | 4437 字的 prompt 可能导致 LLM「认知过载」。尝试限制在 2000 字以内 |
| **分阶段优化** | 🟡 中 | 先单独优化 traceability（独立目标），再单独优化 rigor |
| **多轮优化** | 🟢 低 | 跑 3-5 次完整的 GEPA 优化（每次从上次最佳结果继续），观察是否收敛 |
| **Checker 校准** | 🟢 低 | 当前 checker 过于严格（T≥7+R≥7 才 PASS），可能导致所有 prompt 都无法通过，无法区分优劣 |

### 7.3 对 Self 的即时建议

1. **现在就可以采纳的**：优化 prompt 中的正反例对比格式值得保留——即使整体效果有限，它对特定任务（如技术总结）有显著帮助
2. **不建议现在采纳的**：完整的 4437 字 prompt 过于冗长，建议只提取核心规则+正反例，压缩到 ~1500 字
3. **实验方向**：尝试「短 prompt（核心规则~300字）+ checker 审查后自动注入反思」的混合方案，而非把所有规则都塞进 system prompt

## 8. 决策记录

| 决策 | 级别 | 理由 |
|------|------|------|
| 使用 batch_evaluator 模式而非 per-example evaluator | P2 | adapter 的 evaluate() 天然支持 batch，避免额外 wrapper 开销 |
| acceptance_criterion 设为 improvement_or_equal | P2 | 首次优化，不想过于严格拒绝可能有潜力的候选 |
| reflection LM 用 deepseek-chat 而非 v4-pro | P2 | 成本控制，且 M2 报告中 adapter 已用 chat 模型做评估 |
| 最大 80 metric calls（实际 56）| P2 | 12 次迭代后提案质量持续下降（连续 4 次拒绝），引擎自动停止有意义 |
