## 2. 关键发现

### 2.1 GEPA 未能找到优于种子的候选人

GEPA 完成了8次迭代、49次metric calls（vs 60预算），但**所有候选人的验证集得分 ≤ 种子得分(0.702)**。最优候选人仍为种子prompt(361字符)。

- 迭代1: val=0.677（下降3.6%）
- 迭代2: val=0.655（下降6.7%）
- 迭代3: val=0.676（下降3.7%）
- 迭代5: val=0.701（-0.14%，最接近种子）
- 迭代7: val=0.622（下降11.4%，最差）

Pareto front 从0.715 → 0.735（改善），但**best aggregate**始终卡在0.702。

### 2.2 Phase 4「优化后」测试实为种子复测

因为GEPA未找到更优候选人，Phase 4回到种子prompt进行A/B对比。结果 **0.605 vs 基线0.660 = -8.3%**，但这反映的是SAGE Checker的随机波动（同一prompt两次评测差异达8.3%）。

具体看: Task 2（代码生成进展）的traceability从7暴跌到0，导致该项从0.660降到0.415。这说明**deepseek-chat生成的内容本身存在高方差**，SAGE评分也因此不稳定。

### 2.3 v4-flash 反思质量 vs deepseek-chat

v4-flash生成的proposal具有以下特点：
- 倾向于生成**高度结构化的规则清单**（正反例+格式模板）
- 迭代3的prompt（~1500字）加入了「引用URL必须真实有效（非示例）」等实用约束
- 迭代8的prompt提供了**强制格式JSON-like模板**
- 但所有proposal在val上均未超越种子的0.702

**可能的瓶颈**: 不是reflection LM不够聪明，而是**任务LM(deepseek-chat)遵循prompt改进的能力已达上限**。5条训练数据太少，GEPA的梯度信号不足以引导有意义的方向性改进。

## 4. 根因分析

### 4.1 数据集太小（核心瓶颈）
- **5条训练数据**不足以让GEPA学到可泛化的prompt改进方向
- 每个task的SAGE评分本身有~5-10%的随机波动
- 在5条数据上，0.005的差异≈噪声，GEPA的"信噪比"不足

### 4.2 prompt探索空间不利于缩短方向
- 种子prompt只有361字，GEPA生成的都是1000-1500字的详细prompt
- 更长的prompt在这个任务上并不带来val提升（甚至下降）
- 约束「≤2000字」设得太高——应该设≤500字来迫使精简

### 4.3 任务LM瓶颈
- 当前任务LM（deepseek-chat）在遵守复杂prompt指令方面可能已达性能上限
- R1的4437字大prompt只+1.1%，暗示**prompt长度与SAGE分数之间不是单调关系**
- 更好的reflection LM也无法改善「执行者」的能力上限

### 4.4 Phase 4 方差问题
- 同一prompt两次评测差异达8.3%（0.660 vs 0.605）
- 5条测试数据上，单条traceability从7→0的变化就能拉低整体0.045
- **建议**: A/B测试时每侧至少2次重复取均值

---

## 5. 结论与建议

### 结论
**Round 2 无实质提升，v4-flash升级未解决GEPA在极小数据集上的优化困境。** 优化失败的根本原因不是reflection LM不够好，而是训练信号太弱（5条数据）+ 任务LM已达到prompt响应上限。

### 建议
1. **暂停GEPA在该小数据集上的迭代** — 连跑两轮都未突破，ROI为负
2. **扩大数据集至≥20条** — 目标场景从单一「研究报告」扩大到「知识输出」全类别
3. **改用更强任务LM** — 如 deepseek-v4-pro 或 gpt-5，从执行侧突破
4. **缩短种子prompt** — 从361字进一步压缩到≤150字，给GEPA真正的优化空间
5. **A/B双测取均值** — 每侧至少独立跑2次SAGE评测，避免方差误判
6. **考虑手动优化** — 在极小数据集上，LLM反思不如人类直接改写prompt有效率

### 是否值得推广？
**短期内不宜推广到其他Agent/场景。** 需先在更大数据集上验证GEPA+deepseek组合的可行性后，再决定推广。
