## 根因分析

### 直接原因
Session `agent:main:main` (webchat) 在对话结束后未被正常关闭，OpenClaw 心跳机制持续向其发送轮询，每次都将**全部累积历史**作为上下文发送给 DeepSeek。

### 深层原因

1. **Session 生命周期管理缺失**
   - Webchat session 没有自动过期/关闭机制
   - 心跳轮询不会因"长时间无用户交互"而停止

2. **上下文膨胀无上限**
   - 每次心跳都把全部历史发给模型
   - 380K tokens × 480 次心跳 = 指数级浪费
   - 没有上下文截断或摘要机制

3. **失败重试无熔断**
   - 连续 N 次「assistant turn failed」后没有自动停止
   - 失败仍会计费（token 已在请求中发送）
   - 没有「失败 N 次→暂停轮询」的保护

4. **余额监控缺失**
   - 没有 DeepSeek 余额不足告警
   - 余额耗尽后才由用户发现

5. **多 Agent 归档 Cron 冗余**
   - 7 个 Agent（main/balance/self/xiaofeng）各有独立的午夜记忆归档任务
   - 每个都在做类似的审计工作，总计浪费 ~$0.20/天
   - 即使单个成本小，长期累积可观

## 修复措施

### 立即修复（已完成）
- ✅ 从 `sessions.json` 移除 `agent:main:main` 注册
- ✅ 主 transcript + 轨迹文件 + 43 个备份文件 → `.trash`

### 短期防护（建议）
1. **Session 自动过期**: webchat session >24h 无用户交互 → 自动关闭
2. **心跳熔断**: 连续 3 次失败 → 暂停该 session 的心跳
3. **上下文上限**: 心跳轮询使用摘要而非完整上下文，或限制最大 tokens
4. **余额告警**: DeepSeek 余额 <10 RMB → DM 通知 Daryl
5. **合并归档 Cron**: 7 个 Agent 的午夜归档合并为 1 个，减少冗余成本

### 长期架构（建议）
- Agent 生命周期管理器：统一管理所有 session 的创建/心跳/销毁
- 成本看板实时化：在 OPC 看板显示实时消耗

## 教训

1. **Session = 资源，必须主动管理生命周期**。创建容易，忘记关闭就会变成定时炸弹。
2. **心跳轮询的上下文策略必须优化**。不能把全部历史发给模型——要么用摘要，要么限制长度。
3. **失败不是无害的**。AI API 的失败响应也可能已计费（输入 token 已发送）。连续失败必须有熔断。
4. **自动化系统的成本不可见 = 风险不可控**。余额告警是必需品，不是可选项。
5. **冗余 Cron 在规模下会累积成可观成本**。6/26 引入的多 Agent 归档体系需要重新评估。
