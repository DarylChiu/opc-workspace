## 1. 流程总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ Daryl    │    │ Bryson   │    │ Daryl    │    │ Daryl    │    │ Bryson   │
│ 提出需求 │───→│ 评估方案 │───→│ 确认方向 │───→│ 验收测试 │───→│ 归档文档 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                     │                                 │
                     ▼                                 ▼
               ┌──────────┐                    ┌──────────┐
               │ 如需讨论  │                    │ 发现Bug  │
               │ 多方案对比│                    │ 进入下轮 │
               └──────────┘                    └──────────┘
```

### 核心原则

1. **需求打包迭代** — 多需求统一下发，减少往返沟通消耗
2. **方案先行** — 重大变更先出方案（含A/B对比），Daryl确认后再实现
3. **逐条回复** — 每个需求都要有明确的「原需求→方案→结果」对应
4. **验收即闭环** — Daryl验收通过才算完成，未验收不算闭环
5. **文档同步** — 每轮迭代结束必须更新版本号 + 日记 + active.md

**Bryson方案（每个需求单独回复）**:

- 布局重构确认：body flex row → sidebar(56px) + main-area(flex:1)
- 确认Tab新序：内容总览/脚本解析/镜头声音/复刻模板/标记笔记
- 商业数据确认三平台：B站/抖音/YouTube

**交付**: 当日完成全量5项，index.html重写（~930行），Daryl验收

### R4 · 第一轮Bug修复（06-28 上午）

**Daryl验收发现4个Bug**:

| # | Bug | 根因 | 修复 |
|---|-----|------|------|
| 1 | 🔇 视频无声音 | you-get只选video stream | 下载全部格式+ffprobe检测+ffmpeg合成 |
| 2 | 📜 对白无滚动条 | flex容器缺少min-height:0 | 祖先链全部加flex约束 |
| 3 | 📊 进度条不动 | CSS indeterminate动画 | 改为server端stage-based进度上报 |
| 4 | 💰 商业数据无URL | B站API被境外IP封锁 | 前端直连B站API（浏览器国内IP） |

**关键经验**: 4个bug中有3个是因为「改动了一个点但没拉通全链路」——进度条只改前端没动server、商业数据只测了后端没考虑前端IP差异、视频下载改format没验证音频。

### R5 · 第二轮Bug修复（06-28 下午）

**Daryl复审**:
> 「商业数据之前只加了B站前端fallback，抖音和YT没处理」

**根因**: R4修复只针对B站做了前端fallback，没有扩展到另外两个平台。

| # | Bug | 修复 |
|---|-----|------|
| 1 | 抖音/YT商业数据缺失 | fetchDouyinCommerceFE() + fetchYoutubeCommerceFE() |
| 2 | 视频无声（遗留） | 音频检测→无音频跳过（非直接使用无声视频） |
| 3 | 滚动条太亮 | rgba调整：0.12→0.06, hover 0.25→0.15 |
| 4 | 进度条不显示% | %直接显示在状态栏 + 底部横条 + 客户端估算fallback |

**教训**: 修复bug时必须检查同类问题是否也存在于其他平台/场景。不要修一个漏一片。

### R6 · 抖音下载修复（06-28 晚）

**Daryl验证**:
> 「抖音视频下载又不行了」

**根因**: douyin.com检测海外IP（越南）→ 重定向到首页

**修复**: 
- 发现 iesdouyin.com 海外可访问
- V4下载器：urllib抓取iesdouyin页面 → 正则提取video_id → curl下载
- Playwright降级为fallback
- 测试确认可下载1080p含音频

**关键经验**: 
- 网络问题要区分「代码bug」vs「服务端策略变更」——douyin.com对海外IP的策略和iesdouyin.com不同
- 先搞清楚是哪个环节断的，再看怎么绕
- 不要上来就加复杂方案（Playwright），找简单路径（urllib+正则）

## 3. 角色与职责

| 角色 | 职责 | 典型动作 |
|------|------|----------|
| **Daryl** (产品/验收) | 提出需求、确认方案方向、验收测试、反馈Bug | 飞书消息下达需求 → 看方案 → 跑测试 → 反馈 |
| **Bryson** (开发/归档) | 评估方案、实现代码、自测、文档归档、Git同步 | 分析需求 → 出方案（多方案时A/B对比）→ 实现 → 自测 → 交付 |

### 职责边界

- **Daryl不写的**: 代码、技术方案细节、测试用例
- **Bryson不做的**: 
  - 不替Daryl做产品决策（多方案时必须先请示）
  - 不跳过验收直接标记完成
  - 不自认为修复了bug就完事——必须确认Daryl看到修复结果

## 4. 需求提交流程

### 4.1 Daryl → Bryson

```
Daryl 飞书消息/群聊 → Bryson 逐条解读 → 出方案（多选则A/B对比）
                                            │
                                     ┌──────┴──────┐
                                     ▼              ▼
                               简单需求          复杂需求
                             （UI调整等）     （架构/新功能）
                               直接实现        先出方案确认
```

### 4.2 需求回复模板

Bryson收到需求后的回复格式：

```
收到，逐条回复：

1. [需求1] → 方案：XXX，预计涉及文件：A, B
2. [需求2] → 方案：XXX，有两种思路：
   A方案：XXX（优点/缺点）
   B方案：XXX（优点/缺点）
   请确认方向
3. [需求3] → 简单调整，直接实现，预计5分钟

需要确认：[列出所有需要Daryl拍板的选项]
```

### 4.3 需求优先级

| 级别 | 定义 | 举例 |
|------|------|------|
| **P0 阻断** | 核心功能不可用 | 视频无法下载、分析结果错误 |
| **P1 体验** | 影响使用但可绕过 | 滚动条不显示、进度条不更新 |
| **P2 增强** | 锦上添花 | 商业数据展示、导出格式优化 |
| **P3 低优** | 后续考虑 | 多语言、主题切换 |

## 5. 实现与交付流程

### 5.1 实现前检查清单

```
□ 需求是否完全理解？（有歧义时必须先问）
□ 改动范围是否明确？（涉及哪些文件、哪些函数）
□ 是否需要A/B方案对比？
□ 是否需要先出方案等Daryl确认再动手？
□ 改动是否影响现有功能？（回归风险）
□ 是否需要新增测试/验证步骤？
```

### 5.2 实现中原则

1. **最小改动** — 不要顺手重构不相关的代码
2. **全链路思考** — 改pipeline任一部分要拉通前后环节验证
3. **先对齐目标再搭框架** — 不要为未来可能的需求过度设计
4. **质量优先于体积** — 不要为省空间/省时间牺牲分析质量或功能完整性

### 5.3 交付前自检

```
□ server.py 语法校验通过（python3 -c "import py_compile; py_compile.compile('server.py')"）
□ 本地测试通过（localhost:8777）
□ 版本号同步更新（VERSION + index.html + /api/health）
□ 相关文档更新（如有架构变更→更新DESIGN_REFERENCE.md）
□ Git commit + push
□ 通知Daryl验收（飞书消息）
```

### 5.4 交付通知模板

```
v1.x.x 已部署，改动：

1. [需求1] - 完成，方案：[简述]
2. [需求2] - 完成，方案：[简述]

测试建议：
- 场景A：[具体操作]
- 场景B：[具体操作]

访问：https://xxx.ngrok-free.dev
```

## 6. 验收标准

### 6.1 验收流程

```
Bryson 交付通知
       │
       ▼
Daryl 验收测试
       │
   ┌───┴───┐
   ▼       ▼
 通过     发现问题
   │        │
   ▼        ▼
 归档    Daryl反馈Bug
          │
          ▼
        Bryson分析根因 → 修复 → 重新交付
```

### 6.2 验收通过条件

| 维度 | 标准 |
|------|------|
| **功能** | 所有需求点按方案实现，无遗漏 |
| **可用性** | 正常操作路径可走通，无明显卡顿 |
| **稳定性** | 无崩溃、无500错误、无白屏 |
| **兼容性** | 现有功能未回归（已验证的场景仍正常） |
| **文档** | 版号已更新，关键变更已记录 |

### 6.3 Bug反馈规范

Daryl反馈Bug时，理想信息应包含：

```
1. 操作步骤：[做了什么]
2. 预期结果：[应该看到什么]
3. 实际结果：[实际看到什么]
4. 截图/录屏（如有）
5. 平台：B站/抖音/YouTube
```

**但实际中Daryl可能只给简短反馈**（如「视频没声音」）。Bryson的责任是：
- 不要等Daryl给完整信息再动
- 主动复现问题
- 分析根因而非表面现象
- 修复后检查同类问题是否也存在于其他场景

### 6.4 Bug修复检查清单

```
□ 根因确认？（不是表象）
□ 修复范围检查？（同类问题：其他平台/其他场景/其他文件）
□ 回归测试？（修复是否引入了新问题）
□ 全链路验证？（上下游是否受影响）
□ 通知Daryl复验
```

## 7. 版号管理

### 7.1 语义化版本规则

```
v{major}.{minor}.{patch}

major: 架构级重构（如布局大改、核心管线重写）
minor: 新功能/多需求迭代（如新增商业数据、Sidebar重构）
patch: Bug修复/小优化（如滚动条颜色、进度条调整）
```

### 7.2 版号同步位置

每次变更必须同步更新以下三处：

| 位置 | 文件 | 字段 |
|------|------|------|
| **版号文件** | `VERSION` | 全文 |
| **前端** | `index.html` | `const APP_VERSION` |
| **后端** | `server.py` | `do_GET` 中 `/api/health` 响应 |

> ⚠️ 三处必须一致。不一致会导致前端显示的版本号与后端不匹配。

### 7.3 本项目版号演进

| 版号 | 类型 | 范围 | 触发 |
|------|------|------|------|
| 1.0.0 | major | 初始MVP | 全新项目 |
| 1.1.0 | minor | 5项需求 | Daryl打包需求 |
| 1.1.1 | patch | 4项bug | Daryl验收发现 |
| 1.1.2 | patch | 抖音下载修复 | Daryl验证发现 |

## 8. 经验教训

### 8.1 开发层面

| # | 教训 | 来源轮次 |
|---|------|----------|
| 1 | **不要过早优化** — 为省空间硬编码截断→分析不完整→重做。先对齐目标再搭框架 | R2 |
| 2 | **改动一处要拉通全链路** — 改下载参数没验证音频→视频无声。改一个点要上下游测试 | R4 |
| 3 | **修复要检查同类问题** — 修了B站商业数据没修抖音/YT→第二轮返工 | R5 |
| 4 | **先搞清楚哪个环节断的** — 抖音下载失败不是代码bug而是海外IP策略，找简单路径不要上来就Playwright | R6 |
| 5 | **prompt深度 > 工具选择** — 分析质量问题不是换模型能解决的，是prompt设计不够细致 | R2 |

### 8.2 协作层面

| # | 教训 | 来源轮次 |
|---|------|----------|
| 1 | **需求一次打包下发效率最高** — R3的5项打包比R2的3项零散需求效率高 | R2→R3 |
| 2 | **复杂改动先出方案** — 架构级变更（布局重构、Tab重构）先画方案再写代码，避免返工 | R3 |
| 3 | **Daryl反馈虽简短但精准** — 「视频没声音」一句话背后是下载器参数问题。不要等详细信息，主动排查 | R4 |
| 4 | **验收未通过不算完成** — v1.1.1交付后Daryl发现4个bug，说明自测不够充分 | R4 |
| 5 | **逐条回复+逐条验收** — 打包需求也要逐项确认完成状态，不能笼统说「都做完了」 | R3 |

### 8.3 流程层面

| # | 教训 | 改进措施 |
|---|------|----------|
| 1 | **交付自检不够充分** | 增加交付前自检清单（5.3节） |
| 2 | **bug修复后没系统检查同类问题** | 增加bug修复检查清单（6.4节） |
| 3 | **文档更新滞后于代码** | 版号三处同步强制要求（7.2节） |
| 4 | **缺少迭代记录模板** | 本规范即为此制定 |
