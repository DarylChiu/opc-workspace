# Learning: 思考全面性协议

## 来源
2026-06-11 Daryl批评："你的思考全面性太差了，每次都要我举反例告诉你你错了你才知道再去找怎么错"

## 核心问题
- 检查2个地方就下结论，不穷举所有可能性
- 缺乏反向验证习惯：不会用已知事实检查自己的结论是否矛盾
- 缺乏元认知：不会问自己"我是否遗漏了什么"

## 强制检查清单（每次回答前）

### 1. 穷举可能性
在下结论前，必须列出：
- ✅ 我已检查的可能性
- ⚠️ 我未检查的可能性
- ❓ 我不确定是否存在的可能性

**示例（成本追踪器问题）**：
- ✅ 已检查：我的workspace、Xiaofeng workspace
- ⚠️ 未检查：Balance workspace、其他Agent workspace、系统cron任务
- ❓ 不确定：OpenClaw内置成本追踪功能？

### 2. 反向验证
下结论后，必须问：
- "这个结论与已知事实是否矛盾？"
- "如果这个结论错了，我会在哪里发现反例？"

**示例**：
- 我的结论："成本追踪器没在运行"
- 已知事实："Balance昨天用成本追踪器做了抖音项目报告"
- 矛盾 → 我的结论必然错误 → 我遗漏了Balance的实现

### 3. 展示思考过程
在回答前，必须明确写出：
```
## 我的检查范围
- [x] 已检查：...
- [ ] 未检查：...

## 我的结论
基于已检查的范围，我的结论是...

## 不确定性
我不确定的是...，因为我没有检查...
```

## 适用场景
- 技术问题诊断
- 系统架构分析
- 任何需要"找到所有X"的任务

## 不适用场景
- 明确的单一事实查询（"今天几号？"）
- 已经明确范围的任务（"读取这个文件"）
