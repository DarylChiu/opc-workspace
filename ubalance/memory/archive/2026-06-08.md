## 完整的底稿核心逻辑框架

### **我之前总结的10个维度**

#### **【1. 结构化索引体系】**
- **核心**: 循环+科目的二级编码
- **示例**: 
  - A=投资循环, B=权益变动, C=规划风险, D=收入, E=固定资产
  - H=存货成本 (H1C=Cost of Sales)
  - N=税务 (N1A=Income Tax Expense)
  - R=薪酬 (R1=Employee Benefits, R1-2=Social Insurance Testing)
- **优点**: 任何审计师拿到底稿立刻知道R1-2是R1的下级支撑，项目经理直接按索引定位问题

#### **【2. 重要性水平的全局一致性】**
- **核心**: 每张底稿顶部都标注当年TE (Tolerable Error)
- **示例**: `2021 Component level audit scope: TE at 60% of PM (RMB752,000), trivial at 5% of PM (RMB63,000), 10% fluctuation`
- **优点**: 初级审计师不用翻C6去查TE，强制审计师在每个科目都意识到"这个差异是否超TE"

#### **【3. Lead Schedule + Supporting Schedule分层设计】**
- **三层结构**:
  ```
  R1-LEAD (7张表) → 汇总各公司的工资、社保、公积金
    ↓
  R1-2 For Disclosure (7张表) → 测算vs账面对比，识别差异
    ↓
  R1-2 Reasonableness Test (5+44张表) → 逐月逐人工资明细，深度测算
  ```
- **优点**: 项目经理只看Lead了解整体，复核经理看到差异后深入Supporting找细节

#### **【4. 多版本迭代保留 (Version Control)**
- **三个R1-2版本并存**:
  - R1-2 For Disclosure (披露版，给招股书用)
  - R1-2 Reasonableness Test (测算版，审计证据)
  - R1-2 0117版 (完整版，含2016-2021历史)
- **优点**: 保留审计思路演进轨迹，不同受众看不同版本，避免"改了最终版找不回测算逻辑"

#### **【5. 分类处理的精细化】**
- **示例**: R1-2把不缴社保的原因分6类
  - 正常 / 自愿放弃社保 / 购买新农合及其他国家保险 / 一次性买断社保 / 退休人员 / 实习生
- **优点**: 每类对应不同法律依据，招股书披露时能精确说明"自愿放弃占比X%，已取得书面确认"

#### **【6. Top 5 Disclosure自动化】**
- **核心**: R1-2测算表中直接列示Top 5薪酬
- **示例**: 李忠超 286,000 | 何江 238,660 | 曾秀梅 248,364...
- **优点**: 从底层工资表自动汇总，避免手工统计错误，审计师和客户同时看到

#### **【7. 历史对比的纵深设计】**
- **核心**: 2016-2021年6年数据 (港股IPO只要求3年)
- **优点**: 识别趋势 (社保差异是长期问题还是2021年突发)，说服监管 (证明"一直如此"，不是为IPO突击调整)，风险量化 (6年累计潜在补缴金额)

#### **【8. 实际测试 (Actual Test) 标签】**
- **核心**: 明确区分Planning vs Actual
- **示例**: 工作表名标注"成都2021 (Actual Test)" / "南充2021 (Actual Test)"
- **优点**: 复核时能区分"这是初步估算还是最终审定数"，避免误用Planning数据作为Actual结论

#### **【9. 调整分录追踪体系】**
- **核心**: B9底稿中有AJE#1、AJE#2、CJE#1、CJE#2列
- **示例**: `Per mgt acct | AJE#1 | AJE#2 | AJE#3 | After AJE | CJE#1 | CJE#2 | Audited`
  - AJE = Audit Adjusting Entry (审计调整分录)
  - CJE = Consolidation Adjusting Entry (合并调整分录)
- **优点**: 每笔调整都能追溯到具体编号，客户争议时能快速定位"AJE#3是什么"

#### **【10. 双重质量控制标识】**
- **核心**: 底稿有"编制人"和"复核人"双签名
- **示例**: 编制人: 赵毓钊 | 复核人: 李加圣 | 日期: 2021年7月25日
- **优点**: 责任到人，出问题能追溯，符合审计准则ISQC1的质量控制要求

### **Daryl今天补充的4个要点**

#### **【补充1. 表格之间必有勾稽关系】**
- **核心原则**: 所有表格数字都必须能追溯到上游或下游表格
- **反例**: 国内很多底稿有"孤岛数据"，数字来源不清
- **标准**: Lead Schedule的每个数字都能在Supporting Schedule中找到明细来源

#### **【补充2. 双向视角 — 复核视角 vs 编制人视角】**
- **复核视角** (Top-down，审阅者从汇总看到细节):
  ```
  汇总表Lead → 审定表/披露表 → 明细表&程序执行表Actual test
  ```
- **编制人视角** (Bottom-up，执行者从细节汇总到结论):
  ```
  明细表Actual test → 审定表/披露表 → 汇总表Lead
  ```
- **关键**: 两个视角必须能走通，所有勾稽关系在两个方向都成立

#### **【补充3. H1C Cost of Sales的程序三段式结构】**
- **国内底稿的问题**: 数据过程跟审计程序是脱节的
- **正确的程序记录方式**:
  1. **先写"做什么程序"** (What) — 例如: "对成本明细进行合理性测试"
  2. **然后"做的过程"** (How) — **红色标记** — 合同检查、合理性测试、重新计算等审计程序的具体执行
  3. **最后"该程序执行的结果"** (Result) — 例如: "未发现重大差异" / "识别差异RMB 123,456，已调整"
- **视觉标识**: 把"执行的程序"(How)都做成红色标记，一眼看出审计程序部分

#### **【补充4. Reference链接完整】**
- **核心**: Reference参考要做好链接
- **示例**: 在Excel中，点击Reference能直接跳转到对应的底稿位置
- **优点**: 复核时快速跳转，追踪证据链条不断裂

## 三层架构总结

```
┌─────────────────────────────────────────┐
│  Layer 1: Lead Schedule (汇总表)         │  ← 项目经理/复核层
│  - 总体数字                              │
│  - 调整分录汇总 (AJE/CJE编号追踪)        │
│  - 结论与披露                            │
│  - 复核人签字                            │
└────────────┬────────────────────────────┘
             │ (勾稽关系 — 双向验证)
             ↓
┌─────────────────────────────────────────┐
│  Layer 2: Disclosure/Audited Schedule    │  ← 中间层/披露层
│  (审定表/披露表)                         │
│  - 测算vs账面对比                        │
│  - 差异识别 (标注i/m=immaterial)         │
│  - 调整后数字                            │
│  - Top 5 Disclosure自动化                │
└────────────┬────────────────────────────┘
             │ (勾稽关系 — 双向验证)
             ↓
┌─────────────────────────────────────────┐
│  Layer 3: Supporting/Actual Test         │  ← 执行层/证据层
│  Schedule (明细表&程序执行表)            │
│  - 逐笔/逐月明细                         │
│  - 程序三段式:                           │
│    ① What (做什么程序)                   │
│    ② How (红色标记:合同检查/测算/重算)   │
│    ③ Result (执行结果)                   │
│  - Reference链接到证据                   │
│  - 编制人签字 + Actual Test标签          │
└─────────────────────────────────────────┘
```
