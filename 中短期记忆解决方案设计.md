# Bryson 中短期记忆解决方案设计

## 📌 项目概述
**目标**: 解决Bryson在技术开发过程中的中短期记忆丢失问题，尤其是会话边界记忆断裂和关键决策遗忘
**范围**: 针对案例库中识别的5类问题模式提供系统化解决方案
**时间线**: 短期（1-2周）实施核心功能，中期（1月）完善
**优先级**: 基于案例严重程度排序实施

## 🔍 问题分析（基于案例库）

### 1. 核心问题模式
| 问题类别 | 案例数 | 严重度 | 用户影响 | 技术复杂性 |
|---------|--------|--------|----------|------------|
| STT方案遗忘 | 1 | ☆☆☆☆☆ | 项目停滞 | 高 |
| 关键指令遗漏 | 1 | ☆☆☆☆☆ | 重复沟通 | 中 |
| 服务状态遗忘 | 多次 | ☆☆☆ | 用户体验差 | 低 |
| 技术决策链丢失 | 多次 | ☆☆☆☆ | 技术债务 | 高 |
| 会话边界断裂 | 每天 | ☆☆☆☆ | 效率低下 | 中 |

### 2. 根本原因
- **表层**: 缺乏自动记忆捕获和结构化存储
- **中层**: 现有memory系统设计不适合开发工作流
- **深层**: LLM上下文窗口限制 + 摘要压缩损失

## 📋 具体解决方案

### 方案A：主动式关键信息捕获系统
#### 1. 指令识别引擎
- **触发条件**: 用户消息包含明确动词（"建立"、"修改"、"统计"、"查询"等）
- **提取内容**: 
  - 任务目标
  - 约束条件
  - 截止时间
  - 预期输出
- **存储格式**: JSON结构化任务卡片
- **示例**:
```json
{
  "task_id": "task_2026-04-26_001",
  "user_instruction": "统计Google Cloud STT API使用情况",
  "extracted_requirements": [
    "查询API使用日志", 
    "计算估算消费金额", 
    "提供费用报告"
  ],
  "expected_output": "包含具体数字和费用的清晰报告",
  "deadline": "2026-04-26 18:00",
  "status": "pending",
  "created_at": "2026-04-26 11:17",
  "last_updated": "2026-04-26 11:17"
}
```

#### 2. 技术决策记录系统
- **捕获点**: 
  1. 技术方案讨论（方案A vs 方案B）
  2. 架构决策（端口选择、架构模式）
  3. API选择（Google STT vs Whisper.js）
  4. 编码格式决策（WebM Opus vs LINEAR16）
- **存储结构**:
```
决策链_UID/
├── 决策上下文.md
├── 方案对比.json
├── 最终决策.md
└── 后续验证.md
```

#### 3. 项目状态追踪表
- **追踪对象**:
  - 服务状态（运行/停止/错误）
  - URL/端口配置
  - 部署状态
  - 资源消耗
- **更新机制**:
  - 自动检测（端口扫描）
  - 手动更新（用户确认）
  - 定时汇报（每X小时向用户汇报）

#### 4. 会话边界处理系统
- **会话结束前**:
  - 自动生成"会话总结摘要"
  - 提取"待继续任务"
  - 标记"下一次会话提醒事项"
- **会话开始时**:
  - 自动加载上一次会话的总结
  - 显示待办事项
  - 更新用户: "上次我们讨论了X，现在继续Y"

### 方案B：结构化记忆存储优化

#### 1. 记忆分类系统
```
memory/
├── active/              # 活动中的记忆（当前项目）
│   ├── stt_开发          # 结构化项目文件夹
│   ├── tts_集成          # 每个项目独立
│   └── voice_mvp        # 保持完整性
├── archived/            # 已归档项目
│   ├── voice_mvp_v1.0  # 完整归档
│   └── tts_prototype   # 历史参考
├── decisions/           # 技术决策库
│   ├── 架构决策.json    # 结构化决策
│   ├── api_selection.md # API选择理由
│   └── 编码格式决策.md   # 技术细节
└── state_tracking/      # 状态追踪
    ├── services.json    # 服务状态
    ├── costs.md         # 成本追踪
    └── urls.md          # URL记录
```

#### 2. 跨会话索引系统
- **主索引文件**: `.memory_index.json`
```json
{
  "active_projects": ["stt_开发", "voice_mvp提升"],
  "pending_tasks": [
    {"id": "task_001", "project": "stt_开发", "status": "waiting_for_user", "due": "2026-04-26 18:00"}
  ],
  "last_session_summary": {
    "date": "2026-04-26",
    "main_topics": ["STT API使用统计", "Google Cloud费用报告"],
    "next_steps": ["优化STT响应时间", "集成到Voice MVP"]
  },
  "services_status": {
    "stt_server": {"port": 8096, "status": "running", "url": "https://xxx.ngrok-free.app"},
    "voice_mvp": {"port": 8083, "status": "running", "url": "https://unwhispering-imani-digitately.ngrok-free.dev"}
  }
}
```

### 方案C：用户交互优化

#### 1. 状态主动汇报
- **触发条件**:
  - 用户长时间未互动（>2小时）
  - 关键步骤完成
  - 遇到障碍
- **汇报内容**:
  ```
  📊 进度汇报
  • 当前状态: STT服务器运行中 (端口8096)
  • 最近任务: 完成Google STT长句子测试
  • 待办事项: 统计API使用费用
  • 需要确认: 是否继续优化响应时间？
  • 上次汇报: 已处理2/3的测试用例
  ```

#### 2. 指令确认机制
- **初次接收**: "收到指令: [指令摘要]。我理解需要做[要求]，预计输出[格式]。确认无误请回复'确认'。"
- **任务进展**: "指令执行中: 已完成[进度]，下一步[计划]，预计完成时间[时间]"
- **任务完成**: "指令完成: 结果[摘要]，详细报告[链接]"

## 📊 预期效果指标

### 定量指标
1. **记忆完整率**: 从0%提升到 >80%（关键信息捕获）
2. **重复指令率**: 从 >90%下降到 <20%
3. **用户主动询问状态**: 从每天多次下降到每周1-2次
4. **会话重启时间**: 从5-10分钟下降到30秒内
5. **技术决策追溯**: 从无法追溯提升到100%可查询

### 定性改进
1. **用户体验**: 从"需要不断提醒"到"主动汇报进度"
2. **项目透明**: 从"黑盒状态"到"实时可见状态"
3. **协作效率**: 从"重复沟通"到"高效协同"
4. **信任建立**: 从"怀疑可靠性"到"信赖系统"

## 🔧 技术实现细节

### 文件系统结构
```
~/.openclaw/xiaofeng_workspace/
├── memory/                      # 现有系统（保留）
│   ├── YYYY-MM-DD.md           # 每日原始日志
│   └── heartbeat-state.json    # 心跳状态
├── structured_memory/           # 新增结构化系统
│   ├── active_projects/        # 进行中的项目
│   │   ├── stt_development/
│   │   │   ├── task_cards.json
│   │   │   ├── decisions.json
│   │   │   └── state.json
│   │   └── voice_mvp_enhancement/
│   ├── archived_projects/      # 已完成项目归档
│   ├── user_instructions/      # 用户指令库
│   └── system_status/          # 系统状态记录
├── .memory_index.json          # 系统索引文件
└── state_tracker/              # 实时状态监控
    ├── services_status.json
    └── cost_tracking.json
```

### 关键算法
1. **指令识别算法**: 基于动词列表和模式匹配
2. **会话总结算法**: 提取对话中的决策和待办事项
3. **状态更新算法**: 端口扫描 + 进程检测 + URL验证
4. **记忆检索算法**: 增强的语义搜索 + 时间线索引
