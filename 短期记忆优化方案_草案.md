# 解决上下文压缩导致短期记忆丢失的方案草案

## 问题诊断：上下文压缩（compacted）导致短期记忆丢失

### 根本原因分析
在STT开发过程中暴露的核心记忆问题：
1. **会话边界记忆断裂**：每次会话重启，上下文需要重新建立
2. **关键指令丢失**：跨会话的关键技术决策和指令未能持久化
3. **方案完整性遗忘**：完整的开发方案有时被完全遗忘（如4月15日STT方案）
4. **状态更新滞后**：项目状态变化未能及时更新到长期记忆

### 技术层面分析
1. **LLM上下文窗口限制**：存在固定的token上限（当前64K）
2. **摘要压缩导致的细节损失**：长时间对话被压缩时，技术细节丢失
3. **工作记忆不持久**：短期工作记忆在会话结束后消失
4. **长期记忆触发不完整**：MEMORY.md检索可能不完整或遗漏关键信息

## 目标：提升跨会话记忆完整性的技术解决方案

### 设计原则
1. **非侵入性**：不影响Kitty现有长期记忆系统
2. **可扩展性**：Bryson和Kitty可独立使用或协同使用
3. **自动化**：减少手动记忆操作需求
4. **轻量级**：避免过度复杂的系统设计

## 解决方案提案：分层记忆结构 + 智能触发系统

### 第一层：会话内即时记忆捕获系统
```
功能：实时捕获关键信息和决策点
技术实现：
  - 智能触发词监控：检测"决定"、"同意"、"确认"、"方案"等关键词
  - 自动摘要生成：对话结束后自动生成执行摘要
  - 结构化存储：按项目/任务分类存储到memory/
```

### 第二层：跨会话状态同步系统  
```
功能：确保新会话开始时知道关键状态
技术实现：
  - 会话启动脚本检查：启动时自动检查pending_tasks.md
  - 上次活动摘要：自动展示上次会话的最后活动和状态
  - 状态快照：会话结束时自动保存关键状态快照
```

### 第三层：项目级记忆锚点系统
```
功能：为核心项目建立不可遗忘的里程碑记录
技术实现：
  - 项目建档标准：每个项目强制创建PROJECT_<name>.md
  - 关键决策日志：所有技术决策记录到决策日志
  - 状态检查点：定期更新项目状态，确保可恢复性
```

### 第四层：记忆验证与提醒系统
```
功能：主动验证记忆完整性并提供提醒
技术实现：
  - 每日记忆验证：检查关键项目的记忆状态
  - 缺失提醒：检测记忆空白，主动请求补充
  - 上下文重建工具：基于归档重建完整项目上下文
```

## 具体技术实施方案

### 1. 会话内智能捕获机制（Session-aware Capture）
```python
# 智能关键词触发器
MEMORY_TRIGGER_KEYWORDS = [
    "确认：", "决定：", "方案：", "计划：", "任务：",
    "完成", "失败", "成功", "问题：", "解决：",
    "下一步：", "明天", "需要", "要求"
]

# 每次用户消息后自动运行
def capture_key_decisions(message):
    for keyword in MEMORY_TRIGGER_KEYWORDS:
        if keyword in message:
            # 提取上下文片段
            # 自动记录到memory/
            pass
```

### 2. 会话状态保持器（Session State Persistor）
```yaml
# 会话结束时自动保存到 session_state.json
session_end_state:
  timestamp: ISO时间戳
  last_user_message: 最后用户消息（摘要）
  pending_decisions: 待处理决策列表
  open_tasks: 开放任务列表  
  project_context: 当前项目上下文
  next_steps: 建议的下一步行动
```

### 3. 项目记忆锚点（Project Memory Anchors）
每个核心项目必须包含：
```
PROJECT_<项目名>.md:
  - 项目标识符（唯一ID）
  - 核心目标
  - 关键技术决策（时间线）
  - 当前状态
  - 待办事项
  - 归档文档链接
  - 上次更新的会话ID
```

### 4. 记忆完整性验证器（Memory Integrity Validator）
```bash
# 每日记忆检查脚本
功能：
  1. 检查过去48小时的项目活动记录
  2. 确认关键决策是否归档
  3. 验证项目状态文件是否最新
  4. 提醒缺失的记忆片段
  
频率：每次会话开始时自动运行
```

## 与Kitty现有系统的兼容性设计

### 兼容性保证措施
1. **独立目录结构**：所有新记忆文件存储在 `memory/improved/` 子目录
2. **可选功能**：通过配置文件启用/禁用各项功能
3. **数据隔离**：不修改Kitty现有的MEMORY.md或记忆文件
4. **协同接口**：提供标准的记忆查询API供Kitty调用

### 可选协同模式
1. **独立模式**：Bryson单独使用，完全不影响Kitty
2. **只读协同**：Kitty可读取Bryson的记忆锚点，但不写入
3. **双向同步**：可选的数据同步机制（未来扩展）
